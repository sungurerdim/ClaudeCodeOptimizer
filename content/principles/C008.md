# C008: Breaking Changes Need Approval

## Principle
Never make breaking changes without explicit user approval.

## Why
- Breaks existing functionality
- May affect dependent systems
- Requires coordination with team
- Needs testing and migration plan

## How
- Identify breaking changes before implementing
- Ask user for approval
- Propose migration path
- Document breaking changes clearly

## What Counts as Breaking
- Removing public functions/classes
- Changing function signatures
- Renaming public APIs
- Changing data formats/schemas
- Modifying config file structure
- Database schema changes

## Examples

**Bad:**
```python
# Changed signature without asking
def process_data(data):  # Was: process_data(data, format="json")
    return json.loads(data)
```

**Good:**
```
PROPOSAL:
- Change: Remove 'format' parameter from process_data()
- Reason: Only JSON is supported now
- Impact: All callers need update (5 files)
- Breaking: yes

Should I proceed?
```
